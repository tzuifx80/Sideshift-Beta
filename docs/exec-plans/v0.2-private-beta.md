## Beta user journey

1. Install or open SideShift (PWA or Android APK).
2. Authenticate via email OTP (or deliberate guest continuation).
3. Select a take and start a hosted multi-round SideShift Basic debate.
4. Complete all rounds and receive structured evaluation.
5. Recover honestly from network or provider failures (retry, resume).
6. Close and reopen without losing valid in-progress or completed state.
7. Log out with persistent signed-out marker across WebView restart.
8. Submit beta feedback from settings or debate result.

## Acceptance criteria

- [ ] Production Supabase OTP works with configured email template and manual linking.
- [ ] Deployed Worker serves Basic turn + evaluation with quota enforcement.
- [ ] Authenticated user completes full debate flow on web with real Worker URL.
- [x] Evaluation scores and feedback display without mock mislabelling (code path; live Worker pending).
- [x] Idempotent retry does not double-charge quota or duplicate turns (Worker + turn state tests).
- [x] Offline/provider errors show honest recovery UI, not fake success (auth OTP states, AI debate retry).
- [x] Debate draft and session survive refresh and app restart (persistence code + tests).
- [ ] Android APK uses real HTTPS `VITE_API_BASE_URL` (no placeholder).
- [ ] Android Back, safe areas, logout/restart, and keyboard behave correctly on device.
- [ ] Beta feedback submission reaches configured backend.
- [ ] Release gate passes (`release-gate` skill / `docs/RELEASE_CHECKLIST.md`).

**Out of scope for v0.2:** new major social features, monetization, World Pulse expansion, new league mechanics.

## Manual Cloudflare steps

1. `npx wrangler login`
2. `npx wrangler secret put SUPABASE_URL --env production`
3. `npx wrangler secret put SUPABASE_ANON_KEY --env production`
4. `npx wrangler secret put SUPABASE_SERVICE_ROLE_KEY --env production`
5. Set `ALLOWED_ORIGINS` in `worker/wrangler.jsonc` to production web origin.
6. `npm run api:worker:deploy`
7. `npm run api:worker:verify -- https://<worker-url>`

## Manual Supabase steps

2. Configure OTP email template.
3. Confirm migrations through `0032` applied on production project.

## Security requirements

- No service-role or AI keys in frontend bundle (`vite.config.ts` guards).
- Worker validates bearer on every protected route.
- RLS enforced for all user data mutations.
- CORS limited to configured origins.
- Secret scan clean before release.

## Rollback procedure

1. Revert to previous Worker deployment: `wrangler rollback` or redeploy prior version.
2. Revert frontend/APK to previous build artifact.
3. If migration was applied, use documented down migration or forward-fix only (never edit deployed migrations).
4. Disable new tester access until verified.
5. Record incident in decisions log.

## Known limitations

- Physical Android device testing not completed in last checkpoint.
- `verify:providers:live` may fail on feedback-email delivery assertion.
- Playwright private-social flow blocked by anonymous auth rate limits in prior run.
- Bundle size warnings exist (unrelated to v0.2 gate).
- Subagents (`.cursor/agents/`) require Cursor nightly/beta; frontmatter may strip on stable channel.
- Supabase `saveDebateWithResult` persists the result before marking the debate completed; a debate-save failure can leave an active debate with a saved result until retry (recoverable, not a completed-without-result state).
